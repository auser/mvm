# Plan 130 — Website docs match functionality

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring the website docs (`public/src/content/docs/`) into sync with what the code actually does after the rewrite, **generate** the reference docs that can drift (CLI + SDK), rewrite the prose guides to the new reality, and add a docs-drift gate so a CLI/behavior change can't silently outrun its docs again.

**Architecture:** Two kinds of doc, two strategies. **Reference docs** (the CLI command list, the SDK API) are *generated* from the source of truth — the clap tree and 124's autogenerated SDK types — so they can't drift; a CI check regenerates and diffs. **Prose guides** (mkGuest/Nix, troubleshooting, the contributor guide, positioning) are hand-written and audited against reality, with a gate that compiles/runs their code examples and asserts every `mvmctl <verb>` they mention is real. This complements ADR-066 §9's per-plan rule ("docs change in the same plan that changes a CLI/behavior") with a final sweep + an enforcing gate.

**Tech Stack:** the Astro/Starlight docs site (`public/`), `clap`'s doc emit (`clap-markdown`/`clap_mangen` or `mvmctl gen-docs`), 124's `xtask gen-sdk` (extended to emit the SDK reference), an `xtask check-docs` gate. No new third-party crates in the workspace (the docs generator is build-tooling).

**Prereqs:** the surface-changing plans must land first — 125 (the ≤15 CLI tree), 124/125 (the autogen SDK + `Sandbox`), 123 (backends/storage/mount), 129 (secrets), 122 (encryption). Run **near the end of Stage D**, after the surface settles; the gate wires into CI via 128.

**Why a dedicated plan:** even with §9's per-plan discipline, the rewrite touches every surface, so there is accumulated drift to clear in one audited pass, and the *gate* is a cross-cutting deliverable that no single feature plan owns.

## Phase B — generate the reference docs (no drift)

### Task B1: CLI reference from the clap tree

`reference/cli-commands.md` is hand-maintained today and will be wrong after 125's 52→≤15 restructure. Generate it.

- [ ] **Step 1:** Failing test — `mvmctl gen-docs` (or `clap-markdown` over the `Cli`) emits the full nested command tree (groups + subcommands + flags) as markdown; the committed `reference/cli-commands.md` is that output.
- [ ] **Step 2:** Wire `gen-docs`; replace the hand-written reference with the generated file. A change to a clap command regenerates it. Commit.

### Task B2: SDK reference from the autogenerated types

124's `xtask gen-sdk` produces `sdks/*/_generated/`. Extend it to emit the SDK API reference (types + RPC methods + the `Sandbox` veneer's docstrings) as markdown.

- [ ] **Step 1:** Failing test — `xtask gen-sdk --docs` emits the Python/TS SDK reference; the committed SDK-reference pages are that output.
- [ ] **Step 2:** Wire it; the SDK docs now follow the schema like the SDK code does — one source, no drift. Commit.

## Phase C — rewrite the prose guides

- [ ] **Step 2:** Commit.

### Task C2: the quickstart + the contributor guide + troubleshooting + positioning

- [ ] **Step 1:** Quickstart leads with the `Sandbox` one-call DX (120/125) — `pip install mvm; Sandbox.create(image=…); sb.exec(...)`. `contributing/development.md` reflects the 17-crate layout + the new test tiers (128). `troubleshooting.md` matches the new backends/errors (ADR-053 taxonomy). Positioning leads with the security spine (the gap-analysis differentiators), DX as the floor, and the **local-first** framing — runs on your machine, data never leaves it, no per-minute billing, free + open (the mvm-standalone story; mvmd is the cloud/fleet layer). mvm-vs-mvmd *is* local-vs-cloud.
- [ ] **Step 2:** No competitor names anywhere (repo policy). Commit.

## Phase D — the docs-drift gate

### Task D1: `xtask check-docs`

- [ ] **Step 1:** Failing tests, three checks:
  - **generated-doc no-drift** — regenerate the CLI + SDK reference and assert the committed files match (like the 124 SDK no-drift check).
  - **examples compile/run** — extract fenced code blocks tagged runnable (Rust/Python/`mvmctl`) and compile/execute them; a stale example fails.
  - **referenced-command exists** — scan the docs for `mvmctl <verb> <sub>` and assert each resolves in the clap tree (a doc naming a removed flat verb fails).
- [ ] **Step 2:** Implement `xtask check-docs`; wire into `ci.yml` as a **required** check (via 128). A CLI/SDK change that doesn't update its docs now fails CI. Commit.

## Self-review

- **Coverage:** generated reference (CLI B1, SDK B2), prose rewrite (C1/C2), the enforcing gate (D1), the audit (A1). The split (generate what drifts, gate what doesn't) is the durable answer to "docs match functionality."
- **Grounding:** names the real doc files (`public/src/content/docs/...`), ties each doc to the plan that changed its behavior, and reuses 124's `gen-sdk` + the clap tree as the sources of truth.
- **Honesty:** the gate runs examples, not just lints prose — a doc can't claim a behavior the code doesn't have. No competitor names (repo policy) is an explicit acceptance item.
- **Voice:** comments/notes mark the non-obvious (why generate vs hand-write, why run the examples rather than trust them), not the mechanics.
